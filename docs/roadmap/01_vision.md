# 01. 비전 및 핵심 가설

> Clouvel의 존재 이유와 방향성

---

## 비전

### 단기 비전 (2026)

```
"PRD 없으면 코딩 없다"
- 스펙 없이 AI 코딩 시작하면 차단
- 올바른 방향으로 강제
```

### 장기 비전 (2030)

```
"PRD가 곧 제품이다"
- AI가 코드를 완벽하게 생성하는 시대
- 스펙만 주면 AI가 구현
- "무엇을 만들지"가 유일한 인간의 역할
```

### 비전 진화 경로

```
2026: 스펙 없으면 코딩 차단 (enforcement)
       ↓
2027: 올바른 구조 가이드 (guidance)
       ↓
2028: 팀/조직 수준 스펙 거버넌스
       ↓
2029: 스펙 품질 = 프로젝트 성공률
       ↓
2030: 스펙 → 자동 구현 (AI가 완벽해진 세상)
```

## 철학

### 하지 않는 것 (Never)

```
❌ 코드 생성
   - Clouvel은 코드를 짜주지 않는다
   - 코드 생성은 AI IDE (Cursor, Claude Code)의 역할

❌ 코드 수정
   - 기존 코드를 고쳐주지 않는다
   - 수정도 AI IDE의 역할

❌ 자동 구현
   - 스펙을 주면 자동으로 만들어주는 도구가 아님
   - 구현은 개발자 + AI IDE의 역할

❌ 기술 스택 강제
   - 특정 언어/프레임워크 강요 없음
   - 어떤 기술이든 스펙만 있으면 됨
```

### 하는 것 (Always)

```
✅ Enforcement (강제)
   - 스펙 없이 코딩 시작하면 차단
   - PRD가 없으면 can_code = BLOCK

✅ Guidance (가이드)
   - 올바른 구조/패턴 제안
   - "이렇게 해야 실패 안 함"
   - 흔한 실수 경고

✅ Verification (검증)
   - 스펙 완성도 체크
   - 구현이 스펙과 일치하는지 확인
   - 누락된 요구사항 감지

✅ Templates (템플릿)
   - 프로젝트 유형별 스펙 템플릿
   - 커뮤니티 기여 템플릿
   - 성공 패턴 라이브러리
```

### 철학 요약

```
Clouvel = 코드를 대신 짜주는 게 아니라,
          올바른 방향으로 가도록 강제하는 도구

"Wrong specs + perfect AI = building the wrong thing faster"
잘못된 스펙 + 완벽한 AI = 잘못된 것을 더 빨리 만듦

Clouvel은 "잘못된 스펙"을 방지한다.
```

## 타겟 사용자

### 1차 타겟: Vibe Coders (2026)

```
프로필:
- AI 코딩 도구 적극 사용
- 스펙 없이 "일단 만들어봐" 스타일
- "almost right" 문제 경험
- 기술 부채 누적 중

페인포인트:
- AI가 만든 코드 수정에 시간 낭비
- 프로젝트가 커지면 "tangled mess"
- 무엇을 만들지 명확하지 않음

Clouvel 가치:
- 강제로 스펙 먼저 작성하게 함
- 구조화된 사고 유도
- 실패 방지 가이드
```

### 2차 타겟: 소규모 팀 (2027-2028)

```
프로필:
- 2-10명 개발팀
- AI 도구 도입 중
- 일관성/품질 관리 필요

페인포인트:
- 팀원마다 스펙 품질 다름
- 코드 리뷰 전에 이미 잘못된 방향
- 온보딩 시간 오래 걸림

Clouvel 가치:
- 팀 전체 스펙 표준화
- 스펙 리뷰 워크플로우
- 공유 템플릿
```

### 3차 타겟: 엔터프라이즈 (2028-2030)

```
프로필:
- 50+ 개발자
- 컴플라이언스 요구사항
- AI 도입 거버넌스 필요

페인포인트:
- AI 코드 품질 관리
- 감사 추적
- 보안/규정 준수

Clouvel 가치:
- 조직 수준 스펙 거버넌스
- 감사 로그
- SSO/SCIM
- SOC 2 인증
```
